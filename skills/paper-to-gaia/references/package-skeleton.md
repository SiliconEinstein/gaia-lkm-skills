# Single-Paper Gaia Package Skeleton

This document records the package layout for `$paper-to-gaia` output. Gaia
syntax and metadata shape are governed by the installed Gaia library/CLI;
verify package files with `gaia compile` and `gaia check --hole .` after
emission rather than relying on this template alone.

## Layout

```
<name>-gaia/
  pyproject.toml                  # Gaia knowledge-package metadata
  references.json                 # CSL-JSON record for the source paper
  src/<import_name>/
    __init__.py                   # re-exports + `__all__` (root conclusions only)
    paper_<key>.py                # claims, deductions, optional questions
    priors.py                     # leaf-claim priors (weak points + any isolated conclusions)
  artifacts/paper-extract/
    mapping_audit.md              # per-element transformation log (conclusions, weak points, highlights)
    graph_growth_log.jsonl        # append-only chronological record of every claim/deduction emission, schema-aligned with lkm-to-gaia (see schema below)
    input/<paper>.md              # verbatim copy of the input paper
```

`cross_paper.py` is **not** emitted for single-paper packages.

## Naming Conventions

| Surface                          | Convention                       | Example                            |
|----------------------------------|----------------------------------|------------------------------------|
| Package directory + git repo     | `kebab-case-gaia`                | `liu2015-fibonacci-anyons-gaia`    |
| PyPI / pyproject `name`          | same kebab-case                  | `liu2015-fibonacci-anyons-gaia`    |
| Python import (under `src/`)     | `snake_case`, no `-gaia` suffix  | `liu2015_fibonacci_anyons`         |
| Per-paper module file            | `paper_<key>.py`                 | `paper_liu2015.py`                 |
| Conclusion claim label           | `<key>_c<id>_<suffix>`           | `liu2015_c1_fibonacci_emergence`   |
| Weak-point claim label           | `<key>_c<id>_wp_<suffix>`        | `liu2015_c1_wp_static_screening`   |
| Reference key (CSL)              | `<FirstAuthor><Year>[a-z]?`      | `Liu2015`                          |

Default package name template: `<first-author-surname-lowercase><year>-<short-topic-slug>-gaia`.
Example: paper "Non-Abelian phases in two-component ν = 2/3 fractional quantum
Hall states: Emergence of Fibonacci anyons" by Liu et al., 2015 →
`liu2015-fibonacci-anyons-gaia`.

If the paper is anonymous, has no clear first author, or is missing a year,
fall back to `<short-topic-slug>-gaia` and record the metadata gap in
`mapping_audit.md`.

## `__init__.py` template

```python
"""<name>-gaia — Gaia knowledge package generated from <reference_key>.

Generated by paper-to-gaia. Do not edit by hand for the chain backbone;
add reviewer overrides via priors.py instead.

Source paper: <paper title>
DOI: <paper DOI>
Reference key: <reference_key>
"""
from .paper_<key> import *

__all__ = [
    "<key>_problem",                # Phase 1 motivation as question(...)
    "<key>_c1_<suffix>",
    "<key>_c2_<suffix>",
    # ...one entry per Phase 1 conclusion...
    # "<key>_open_question_1",     # only if open questions are emitted (opt-in)
]
```

Public symbols are: the paper's motivation `question(...)`, every Phase 1
conclusion `claim(...)`, and any opt-in open-question `question(...)`
nodes. Weak-point claims are imported internally (for use in deductions
and priors) but **not** re-exported.

## Python LaTeX-string convention

Claim bodies and `deduction(...)` `reason=` fields contain LaTeX math
delimited by `$...$`. LaTeX uses backslashes (`\frac`, `\mathrm`, `\sigma`,
`\\`), and Python's default string literal interprets backslash as an
escape — `"\frac"` becomes `frac` with an embedded form-feed-like char,
not the literal `\frac` LaTeX needs. To avoid this, wrap **every** body or
`reason=` string that contains LaTeX in a **raw** string:

```python
<label> = claim(
    r"In the 2D Yukawa Fermi gas at $T = 0.04 T_F$ and dimensionless "
    r"coupling $\lambda / k_F = 0.5$, the static polarization "
    r"$P(q=0,\omega=0)$ obtained from the perturbative expansion ...",
    claim_kind="conclusion",
    ...
)

deduction(
    [...],
    <conclusion_label>,
    reason=r"""
1. The Hamiltonian $H = \sum_k \epsilon_k c^\dagger_k c_k + \frac{1}{2V} \sum ...$
   describes ...
2. ...
""".strip(),
    prior=0.95,
)
```

The `r` prefix turns off backslash interpretation, so `\frac` survives as
the literal four characters `\frac`. For multi-line text the same applies
to triple-quoted raw strings (`r"""..."""`). Plain (non-LaTeX) bodies do
not need the `r` prefix, but using it consistently is harmless and
avoids correctness regressions when LaTeX is added later.

Backslash-newline-substitution intent (`\\` meaning a LaTeX line break)
still works: in an `r"..."` string `r"\\"` produces the literal two
characters `\\`, which is what LaTeX expects.

Non-LaTeX bodies — typical in clinical, social-science, or systems
papers — follow the same self-containment discipline but do not need
math delimiters or a raw-string prefix:

```python
<label> = claim(
    "Faricimab administered as 6.0 mg intravitreal injections every 16 "
    "weeks after a four-monthly loading phase produces a mean change "
    "from baseline best-corrected visual acuity (BCVA) of +11.4 ETDRS "
    "letters at week 52 in the study eye, with a mean of 6.2 injections "
    "through week 52, in treatment-naive participants with neovascular "
    "age-related macular degeneration; BCVA is reported as ETDRS letter "
    "score change from baseline where higher positive values indicate "
    "improved visual acuity.",
    claim_kind="conclusion",
    source_paper="<reference_key>",
    provenance_source="paper_extract",
    review_prior=0.82,
)
```

The body inlines the regimen, the outcome metric, the population, the
follow-up duration, and the metric direction — a reader unfamiliar with
the paper can identify all of these from the body alone, exactly as the
self-containment rule demands. The same shape applies whether the body
is heavy in LaTeX (theoretical / experimental physics, math, signal
processing) or pure prose (clinical RCTs, behavioral experiments,
qualitative case studies); only the LaTeX-handling convention above
differs.

## `paper_<key>.py` template

```python
"""paper_<key> — claims and deductions for <paper title>.

Source: <paper.title>
DOI: <paper.DOI>
Authors: <paper.authors>
Reference key (CSL): <key>
"""
from gaia.lang import (
    claim, question,
    deduction,
)

# --- Motivation as question(...) ------------------------------------------
# Aligned with pipeline B (XML→LKM): <problem> in select_conclusion.xml
# becomes a LocalVariableNode(type="question"). See mapping-contract.md §5.

<key>_problem = question(
    r"<self-contained motivation text — pre-paper problem state>",
    source_paper="<key>",
    provenance_source="paper_extract",
)

# --- Conclusions -----------------------------------------------------------

<key>_c1_<suffix> = claim(
    r"<self-contained conclusion text — see mapping-contract.md §1>",
    claim_kind="conclusion",
    source_paper="<key>",
    provenance_source="paper_extract",
    review_prior=0.78,                    # Phase 3 conclusion-synthesis prior; capped at 0.9
    refs=(
        {"type": "figure", "id": "Fig. 3"},
        {"type": "citation", "key": "Smith2020"},
    ),
)

# ... one claim() per Phase 1 conclusion ...

# --- Weak-point leaf claims ------------------------------------------------

<key>_c1_wp_<suffix> = claim(
    r"<self-contained weak-point text>",
    claim_kind="weak_point",
    source_paper="<key>",
    provenance_source="paper_extract",
    weak_types=("model",),
    p1=0.7,
    p2=0.85,
    refs=(
        {"type": "equation", "id": "Eq. (12)"},
    ),
)

# ... one claim() per Phase 3 weak point ...

# --- Deductions ------------------------------------------------------------

deduction(
    [
        # direct upstream conclusions from Phase 1 logic graph (if any) ...
        # weak-point claims for this conclusion (if any) ...
    ],
    <key>_c1_<suffix>,
    reason=r"""
1. <Phase 2 step 1 prose>.
2. <Phase 2 step 2 prose>.
""".strip(),
    prior=0.95,                       # warrant strength of THIS chain, NOT the
                                      # conclusion's posterior; see mapping-contract.md §3.
                                      # Adjust per highlight (+0.02..+0.04) / per gap
                                      # (-0.05..-0.10) and record adjustments in audit.
)

# ... one deduction() per derived conclusion ...

# --- Open questions (optional; opt-in only) -------------------------------
# Pipeline B does not extract open questions; emit only on user request.
# If emitted, label format `<key>_open_question_<n>` and add to __all__.

# <key>_open_question_1 = question(
#     r"<self-contained open-question text>",
#     source_paper="<key>",
#     provenance_source="paper_extract",
# )
```

## `priors.py` template

```python
"""priors.py — leaf-claim priors for this package.

Generated by paper-to-gaia. Every entry is auto-seeded and TODO-marked. Run
`gaia check --hole .` to surface any leaves that did not get an entry.
"""
from .paper_<key> import (
    <key>_c1_wp_<suffix>,
    # ... one import per leaf claim ...
)

PRIORS = {
    <key>_c1_wp_<suffix>: (
        0.65,
        "model assumption — static screening adopted without ab initio "
        "verification at the studied density; TODO:review",
    ),
    # ... one entry per leaf claim ...
}
```

Cap priors at 0.9. Use 0.001 as the lower bound (Cromwell). Every
justification text must end with `TODO:review`.

## `references.json` template

```json
{
  "<reference_key>": {
    "type": "article-journal",
    "title": "<paper title>",
    "DOI": "<paper DOI>",
    "container-title": "<journal name>",
    "issued": {"date-parts": [[<year>]]},
    "author": [
      {"family": "<surname>", "given": "<given names>"}
    ]
  }
}
```

If the paper Markdown does not carry author / DOI / year, leave the field
absent (do not invent values) and record the gap in `mapping_audit.md`.

## `artifacts/paper-extract/` Contents

- `mapping_audit.md` — flat per-element decision log (see template below).
- `graph_growth_log.jsonl` — append-only chronological record of every
  emitted claim, deduction, and prior. Schema-aligned with the lkm-to-gaia
  ecosystem's `timeline-log-contract.md` (`schema_version: "1"`) so a
  frontend can replay the Gaia starmap from `t=0` without parsing the
  Python source. paper-to-gaia uses a subset of the contract (no retrieval
  events because there is no LKM API; no frontier rounds because the
  workflow is single-pass). See "graph_growth_log.jsonl schema" below.
- `input/<paper>.md` — verbatim copy of the input paper, so the package is
  reproducible without the original input directory.

The directory is named `paper-extract/` (not `lkm-discovery/` like
lkm-to-gaia) because paper-to-gaia's upstream is a single paper Markdown
file, not LKM. The two skills produce different audit-dir names but use
the **same `graph_growth_log.jsonl` schema** — frontend replay tooling
that locates the log by glob (`**/graph_growth_log.jsonl`) reads both
without modification. Files unique to LKM workflows
(`retrieval_log.jsonl`, `merge_audit.md`, `merge_decisions.todo`,
`dismissed/`) are simply absent from paper-extract packages.

## `mapping_audit.md` template

```markdown
# Mapping audit log — <package name>

Source paper: <reference_key> ("<paper title>")

## Phase summary

| phase | counts |
|---|---|
| Phase 1 conclusions | <n> |
| Phase 1 logic-graph edges | <n> |
| Phase 2 reasoning steps (total) | <n> |
| Phase 3 weak points | <n> |
| Phase 3 highlights | <n> |
| Phase 1 motivation block | yes / no |
| Phase 1 open-question block | yes / no |

## Conclusions

| label | conclusion_id | title | upstreams | weak_points | deduction_prior | review_prior | notes |
|---|---|---|---|---|---|---|---|
| <key>_c1_<suffix> | 1 | ... | (none) | wp_static_screening, wp_finite_size | 0.95 | 0.78 | ... |

`weak_points` lists the weak-point labels bound to this conclusion (per
`mapping-contract.md §2a`, each weak point is bound to exactly one
conclusion). The `notes` column records any deduction-prior adjustments
and their rationale (highlight underwriting, flagged gaps).

## Weak points

| label | conclusion_id | also_threatens | weak_types | prior | p1 | p2 | weakness_reason | failure_mode |
|---|---|---|---|---|---|---|---|---|
| <key>_c1_wp_<suffix> | 1 | (none) | model | 0.65 | 0.7 | 0.85 | <Phase 3 reason text> | <Phase 3 failure_mode text> |
| <key>_c2_wp_<suffix> | 2 | 4 | model, statistical | 0.55 | 0.6 | 0.8 | ... | ... |

`conclusion_id` is the single conclusion this weak point threatens — the
weak point appears as a premise in only that conclusion's deduction, per
`mapping-contract.md §2a`.

`also_threatens` is **audit-only**: a comma-separated list of independent
conclusion ids whose derivations also rest on the same scientific
assumption but to which the weak point is **not** wired in BP (it would
otherwise double-count the assumption as independent evidence). Use
`(none)` when the assumption is specific to the bound conclusion.

The claim body lives in `paper_<key>.py`; the table here carries the
reviewer critique fields that do not belong inside the DSL.

## Highlights

| highlight_id | conclusion_id | strength_types | credit | notes |
|---|---|---|---|---|
| H1 | 1 | computational, statistical | <Phase 3 credit text> | influenced deduction prior (0.95 → 0.97) |

## Motivation (paper-level)

<Phase 1 motivation text, copied verbatim>

## Open questions (paper-level)

<Phase 1 open-question text, copied verbatim>

## Metadata gaps and rationale

- <e.g. "DOI not present in the input Markdown — left absent in references.json">
- <e.g. "Phase 1 detected only 1 atomic conclusion; the paper's secondary claim was a refinement and was filtered as non-novel">
```

## `graph_growth_log.jsonl` schema

The schema mirrors the lkm-to-gaia ecosystem's
[`timeline-log-contract.md`](https://example.invalid) so existing replay
tooling can read paper-to-gaia packages without modification. paper-to-gaia
emits the **graph-growth** subset only — there is no `retrieval_log.jsonl`
because no LKM API calls are made.

### Common event-identity fields

Every event carries:

```json
{
  "schema_version": "1",
  "event_id": "<timestamp_ms>__<actor_id>__<channel_or_decision>__<seq>",
  "timestamp_utc": "YYYY-MM-DDTHH:mm:ss.SSSZ",
  "stage": "mapping",
  "round_id": "round_0000",
  "actor": "paper-to-gaia",
  "actor_id": "paper-to-gaia-<pid_or_uuid>",
  "seq": <int>,
  "phase": 4
}
```

paper-to-gaia-specific conventions:

- **`stage`** is always `"mapping"` (the workflow is one-shot mapping; there
  is no cold-start / frontier-expansion phase).
- **`round_id`** is always `"round_0000"` (no multi-round expansion).
- **`actor`** is always `"paper-to-gaia"`.
- **`actor_id`** is included in **every** event's `event_id` (not only the
  init event). Use the format
  `<timestamp_ms>__<actor_id>__<channel_or_decision>__<seq>` consistently.
- **`<channel_or_decision>`** is the literal string from the `decision`
  field of the same event (`package_initialized`, `accepted_claim`,
  `accepted_deduction`, `prior_added`, `obligation_added`), with one
  exception: the `package_initialized` event uses the shorter token
  `init` for readability. No other shortenings.
- **`phase`** (extension field) is fixed at `4` — paper-to-gaia writes
  every event during Phase 4 emission, even though the analytical
  decisions were *made* in Phases 1–3. The `phase` field captures emit
  time, not analytical-discovery time. If a consumer needs the
  analytical phase, put it in the event's `notes` field. Consumers built
  for lkm-to-gaia will ignore the extra field.
- **`seq`** is a monotonically increasing integer per actor; reset is not
  allowed within a run.
- **Logical order, not wall-clock order.** All events are written at the
  end of Phase 4 in a single batch, but `seq` and `event_id` reflect the
  **logical emission order** (init → all conclusion claims in
  topological order → all weak-point claims grouped by conclusion → all
  priors → all deductions in topological order). A frontend that replays
  by `seq` will see the package come into existence in the order Phase 4
  conceptually emitted it, regardless of when the file was actually
  flushed to disk.

### Allowed `decision` values

paper-to-gaia uses this subset of the lkm-to-gaia decision vocabulary:

- `package_initialized` — first event; payload includes the package name
  and `source_paper` reference key.
- `accepted_claim` — for every conclusion-claim and weak-point-claim
  emitted in `paper_<key>.py`. Distinguish via `payload.claim_kind`.
- `accepted_deduction` — for every `deduction(...)` emitted. Edges in
  `graph_delta.edges_added` are one per premise → conclusion pair.
- `prior_added` — for every entry in `priors.py`. payload: `label`,
  `prior`, `justification`.
- `obligation_added` — emit only when the user has explicitly asked
  paper-to-gaia to materialize Phase 1 open-questions as `question(...)`
  nodes.

These are all defined in lkm-to-gaia's contract, so a replay tool will
treat them identically.

### Mandatory rules

- `graph_delta` is required on every event. Empty arrays are valid for
  events that change priors-only or audit-only.
- Allowed node `kind` in `graph_delta.nodes_added`: `claim` and
  `question` (the latter for the motivation node and any opt-in
  open-question nodes — see `mapping-contract.md §5`). `deduction`,
  `support`, `contradiction`, `equivalence` are valid in lkm-to-gaia
  but not normally emitted by paper-to-gaia for single-paper packages.
- Allowed edge `kind`: `deduction` (the only relation paper-to-gaia
  normally emits for single papers; `support`, `contradiction`,
  `equivalence` are reserved for cross-paper merges).
- **One `accepted_deduction` event per `deduction(...)` call**, even when
  the deduction has multiple premises. The event's `payload.premises` is
  the full list and `payload.reason_excerpt` carries the truncated
  `reason=` prose **once**, not per edge. `graph_delta.edges_added` lists
  one slim edge per `(premise → conclusion)` pair, carrying only `from`,
  `to`, `kind`, and `prior`. Do **not** repeat `reason_excerpt` on each
  edge — it is per-deduction, not per-edge.
- `gaia_actions[].symbol` is the Python identifier the call binds to; for
  `deduction(...)` calls (which Gaia does not return as a named symbol)
  use `null`. This is intentional and stable; consumers should treat
  `symbol: null` as "anonymous strategy" rather than as missing data.
- `audit_files` must point to `mapping_audit.md` so the audit row remains
  reachable from any structured event.
- Append-only: never rewrite an earlier line. Mistakes get a later
  correction event with `supersedes_event_id` referencing the original.
- The `notes` field is allowed on any event for short rationale that does
  not fit elsewhere. Typical uses:
  - On `accepted_claim`: which Phase 1 logic-graph edge first surfaced
    this conclusion (`"Phase 1 c2 → c4 edge"`), or the analytical phase
    that discovered a weak point (`"Phase 3 weak-point gating; pattern:
    formal"`).
  - On `accepted_deduction`: the per-highlight / per-gap prior
    adjustments behind the warrant (`"baseline 0.95; +0.04 H2; -0.05
    flagged-gap on q-extension; net 0.94"`).
  - On `prior_added`: the calibration band the prior was placed in
    (`"heuristic-extrapolative band 0.40-0.60"`).
  Keep `notes` short (≤ 200 chars). Anything longer belongs in
  `mapping_audit.md`.

The detailed scientific rationale (full `weakness_reason`, `failure_mode`,
`credit`, narrative) lives in `mapping_audit.md`. The JSONL is the
ordered structured index that lets a frontend replay the workflow without
parsing markdown.
