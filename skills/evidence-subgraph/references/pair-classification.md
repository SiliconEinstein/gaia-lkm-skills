# Pair classification (contradictions and cross-validation)

This file is the canonical specification for how the discovery flag pass — defined in `$evidence-graph-synthesis` §2b — decides whether a candidate pair is **promoted** to `contradictions.json` / `cross_validation.json`, or **dismissed** to `dismissed_pairs.json`. The schema for each file is fixed by `run-folder-output-contract.md`. This file fixes the **judgement criteria** that drive the verdict.

The classification is the same regardless of the user's prompt language. The agent applies it to LKM payload content (`content` fields plus `data.papers` metadata) using semantic judgement.

## 1. Scope

Two pair types pass through this classification:

- **Contradiction candidates** — two source-grounded claims that, on the surface, assert conflicting values, signs, scaling regimes, or directional outcomes for a comparable target.
- **Cross-validation candidates** — two source-grounded claims that, on the surface, agree (or partially agree / partially disagree) on a comparable target via different methods or measurements.

Equivalence pairs (same proposition, same target) are out of scope here — they are written directly to `equivalences.json` with lineage classification per `$evidence-graph-synthesis` §2b. If a candidate first looked like a contradiction but turns out to assert the same proposition, dismiss it as `confirmed_equivalence` (see §6) and, when the lineage is informative, additionally record it in `equivalences.json`.

## 2. Contradiction — operational definition

A **scientific contradiction** is a non-trivial tension between two source-grounded claims, comparable in context, that cannot be dissolved by an obvious difference in conditions and that has the potential to seed a new explanation, mechanism, boundary condition, or research question.

Three properties are jointly necessary:

1. **Source-grounded.** Both sides come from LKM payload content with locatable provenance (`claim_id` + RFC 6901 pointer into `raw/`). Source-grounding is a **pre-classification gate**, not a verdict: a candidate that lacks locatable provenance on either side is dropped at discovery time (see `$evidence-graph-synthesis` §2b) and never becomes a pair record. Such drops are not written to `dismissed_pairs.json` — there is no verdict for them, because there is no record.
2. **Context-comparable.** The two claims target the same observable in conditions a domain reader would judge mutually relevant. If the conditions visibly differ (different temperature regime, different sample, different model assumption, different measurement protocol) **and** that difference already has a known mediating explanation in the field, the tension is not a contradiction — see `resolved_moderator` in §6.
3. **Generative.** The tension can be articulated as a checkable new question that, if pursued, would produce new knowledge (a hidden variable identified, a boundary mapped, a protocol cleaned up, a model assumption surfaced, an evidence-reliability problem flagged). A surface conflict that does **not** generate such a question is not a contradiction in this skill's sense — see `non_generative` in §6.

A pair that survives the pre-classification gate (property 1) but fails property 2 or property 3 is **not promoted**. It is recorded in `dismissed_pairs.json` with the appropriate verdict.

## 3. Contradiction — promotion gate

Before writing a pair to `contradictions.json`, the agent must articulate, in one sentence, a **checkable new question** generated by the tension. If the agent cannot, the pair is dismissed.

A checkable new question typically has the form: *"Under what conditions does X hold vs Y?"* / *"Which hidden variable distinguishes the two regimes?"* / *"Does protocol P bias the result toward X?"* / *"Is the model assumption A masking the true mechanism?"* / *"Is one side's evidence chain reliable?"*

Generic gestures ("more work is needed", "the field should investigate") do **not** count. The new question must name the specific axis along which a future measurement or calculation would discriminate.

## 4. Contradiction — required structured fields

Every record in `contradictions.json` carries:

- `verdict: "promoted"` — fixed value; explicit type-tag for the self-check.
- `new_question` — the one-sentence checkable question generated by the tension. Must be non-empty.
- `hypothesized_cause` — array of one or more values from the closed enum in §5. Multi-select is allowed; `other` is not allowed; the empty array is not allowed.
- `rationale` — one to two sentences explaining why the tension cannot be dissolved by an obvious difference in conditions. (Distinct from `new_question`: `rationale` defends "this is a real tension"; `new_question` says "and here is what to do with it".)

Plus the common pair fields (`id`, `claims[]`, `drawn_in_graph`, `graph_pair_edge_id`) defined in `run-folder-output-contract.md` §6.

## 5. Hypothesized-cause taxonomy (closed, multi-select, no `other`)

Each value identifies one mechanism class through which the tension could plausibly be resolved by future work. Multi-select reflects honest uncertainty: when several causes are plausible, list all that apply. Do not invent values; expansion of this enum is a contract change and must be made in this file first.

| value | meaning |
|-------|---------|
| `hidden_variable` | A third variable, not yet recognised in the comparison, may distinguish the two regimes. Resolving the tension would identify and characterise it. |
| `boundary_condition` | The two claims hold in different parts of a parameter space whose boundary has not been systematically mapped. Resolving the tension would map that boundary. |
| `measurement_protocol` | The two claims use measurement / data-acquisition / preprocessing protocols that are not equivalent in ways that have not been audited. Resolving the tension would isolate and quantify the protocol effect. |
| `model_assumption` | The two claims rest on different theoretical / computational assumptions (symmetry, closure, prior, parameterization) whose impact on the target has not been disentangled. Resolving the tension would surface the masking assumption. |
| `evidence_reliability` | One or both sides may have a problem at the level of evidence quality (sample bias, low statistics, replication gap, undisclosed prior, validation gap). Resolving the tension would re-test under stricter evidence standards. |

This taxonomy is also the structuring axis for the `$scholarly-synthesis` §6 (open problems) narrative: each promoted contradiction's `hypothesized_cause[]` selects which hypothesis sub-paragraph the open question lives in.

## 6. Contradiction verdict taxonomy (dismissals)

When the promotion gate fails, the agent assigns one of the following verdicts and writes the record to `dismissed_pairs.json` with `origin: "contradiction"`. Verdicts are mutually exclusive — pick the closest match; if more than one applies, pick the one closest to the **first** disqualifying property in §2.

| verdict | when to use |
|---------|-------------|
| `confirmed_equivalence` | On closer reading the two claims collapse to the same underlying work or the same proposition on the same target. This includes: arXiv preprint vs journal version of the same paper (detect via DOI prefix `10.48550/arxiv.` and `area: "arXiv-…"` against a matching non-arXiv DOI / journal entry, with matching titles or author lists), mis-deduplicated id lineage, same-author republication or self-quote, and apparent conflicts that on closer reading are wording or rounding artefacts of the same proposition rather than differences in content. The pair may additionally be recorded in `equivalences.json` with appropriate lineage classification. |
| `resolved_moderator` | The two claims target conditions that visibly differ along an axis the field already explains with a known mediator. The "conflict" is the expected condition-dependence and is not generative. Example: one rate constant at low pressure, another at high pressure, with the pressure dependence already characterised in the literature. |
| `non_generative` | The pair survives both the same-work check and the resolved-moderator check — it is a real surface conflict between two distinct works on a comparable observable — but the agent cannot, in good faith, articulate a checkable new question naming a specific axis a future measurement or calculation would discriminate (property 3 of §2). The agent has tried and the question does not crystallise. Use this verdict instead of forcing the pair into a verdict that does not actually fit. The `rationale` should name what the agent attempted (which `hypothesized_cause` candidates were considered, why none yielded a sharp question). Future runs may revisit such pairs with more context. |

A dismissed-as-contradiction record carries `verdict` and `rationale` (explaining why the dismissal applies) but **not** `new_question` or `hypothesized_cause`.

**Out of scope: corpus-layer artefacts.** Apparent conflicts that the agent suspects come from upstream corpus construction (extraction-pipeline error, unit-conversion mistake at extraction time, sign-flip in a paraphrased figure caption, etc.) are **not** a verdict in this skill. The agent cannot reach outside the chain payload to verify such suspicions, and a verdict that depends on subjective extraction-quality judgement would not be reliably applicable. When the agent suspects a corpus artefact but has no evidence beyond the suspicion itself, the pair must still go through the promotion gate; the agent may flag the suspicion in `rationale` (e.g. "the apparent disagreement could be an extraction artefact, but the chain payload provides no internal evidence either way") for human review, but must not silently dismiss the pair on that basis. Genuine corpus-construction problems are surfaced upstream to the LKM service, not encoded as verdicts here.

## 7. Cross-validation — operational definition

A **scientific cross-validation** is a non-trivial agreement (or partial agreement / partial disagreement) between two source-grounded claims that target the same observable through methods, measurements, or pipelines whose dependencies are independent enough that the agreement actually increases scientific confidence in the target — or, in the partial-disconfirm polarity, that the disagreement actually points to a meaningful open question.

Three properties are jointly necessary:

1. **Source-grounded.** Same pre-classification gate as §2 — candidates lacking locatable provenance on either side are dropped at discovery time and never become pair records.
2. **Comparable target.** The two claims address the same observable in mutually relevant conditions.
3. **Independence of pathway.** The two pathways do not share enough construction that agreement would be forced. Independence may live in any of: independent samples, independent measurement instruments, independent theoretical formalisms, independent simulation codes, independent datasets, independent parameter calibrations.

A pair that survives the source-grounding gate but fails property 3 is dismissed as `trivially_dependent` (see §10) — agreement provides little additional confidence because the two pathways are not actually independent. A pair that satisfies all three properties but on which the agent cannot articulate non-empty `independence_basis` and `scientific_weight` (the §8 promotion gate) is dismissed as `non_generative` (see §10).

## 8. Cross-validation — promotion gate

Before writing a pair to `cross_validation.json`, the agent must articulate, in one sentence each:

- **`independence_basis`** — what specifically makes the two pathways independent (e.g. *"independent measurement on different sample lots with cross-instrument calibration"*, *"first-principles calculation vs phenomenological fit"*, *"experimental observation vs ab-initio simulation"*).
- **`scientific_weight`** — what the agreement (or partial disagreement) actually buys: increased confidence in which specific aspect of the target, or what new question the partial disagreement opens. Generic gestures ("confirms the result", "supports the literature") are not sufficient.

If the agent cannot articulate both, the pair is dismissed.

## 9. Cross-validation — required structured fields

Every record in `cross_validation.json` carries:

- `verdict: "promoted"` — fixed value; type-tag.
- `polarity` — one of `confirm | partial_confirm | partial_disconfirm`. Existing field, semantics unchanged.
- `independence_basis` — non-empty sentence defined in §8.
- `scientific_weight` — non-empty sentence defined in §8.
- `rationale` — one to two sentences summarising why the pair is non-trivial cross-validation rather than a trivial dependency or a re-statement.

Plus the common pair fields per `run-folder-output-contract.md` §6.

## 10. Cross-validation verdict taxonomy (dismissals)

When the cross-validation promotion gate fails, the agent writes to `dismissed_pairs.json` with `origin: "cross_validation"` and one of:

| verdict | when to use |
|---------|-------------|
| `confirmed_equivalence` | The two sides are not actually two separate reports — they are restatements of the **same underlying work** (same paper, arXiv preprint vs journal version of the same paper, the same author republishing or self-quoting the same result, mis-deduplicated id lineage). There are not two pathways at all; there is one pathway counted twice. |
| `trivially_dependent` | The two sides **are** two pathways (different reports, possibly different authors), but the pathways share enough upstream construction (same dataset, shared upstream calculation, derived from the same model under different post-processing, sibling re-analyses of the same observation) that the agreement is by construction or near-by-construction. The pair fails property 3 of §7 even though it survives the same-work check. |
| `non_generative` | The pair survives both the same-work check and the independence check — it is two distinct works on a comparable observable through pathways the agent judges plausibly independent — but the agent cannot, in good faith, articulate non-empty `independence_basis` or non-empty `scientific_weight` (the §8 promotion gate). The agent has tried and the substance does not crystallise. Use this verdict instead of forcing the pair into a verdict that does not actually fit. The `rationale` should name what the agent attempted and where it stalled. |

The three verdicts have a strict precedence enforced by the decision procedure in §12: `confirmed_equivalence` (same work counted twice) is checked before `trivially_dependent` (distinct works whose pathways share construction), which is checked before `non_generative` (independent pathways whose substance the agent could not articulate).

**Out of scope: corpus-layer artefacts.** As in §6, suspected corpus-construction problems (extraction error, unit-conversion mistake, sign-flip during ingestion, etc.) are not a verdict here. The agent cannot reach outside the chain payload to verify such suspicions. When suspected, flag in `rationale` for human review; do not dismiss on that basis alone.

A dismissed-as-cross-validation record carries `verdict` and `rationale`, but **not** `independence_basis`, `scientific_weight`, or `polarity`.

## 11. Verdict allowability matrix

The `verdict` value in `dismissed_pairs.json` must be compatible with the record's `origin`:

| origin | allowed verdicts |
|--------|------------------|
| `contradiction` | `confirmed_equivalence`, `resolved_moderator`, `non_generative` |
| `cross_validation` | `confirmed_equivalence`, `trivially_dependent`, `non_generative` |

The self-check (`run-folder-output-contract.md` §8) enforces this matrix.

## 12. Decision procedure (when comparing a candidate pair)

**Pre-classification gate (both origins).** Before any verdict is considered, the candidate pair must satisfy property 1 (source-grounded) — both sides have locatable provenance (`claim_id` + RFC 6901 pointer into `raw/`). Candidates that fail this gate are **dropped at discovery time** and never become pair records; they are not written to any pair file. There is no verdict for non-source-grounded candidates.

For each candidate that passes the pre-classification gate, walk the questions below in order. The first triggering condition determines the verdict.

For a contradiction candidate:

1. Do the two sides collapse to the same underlying work or the same proposition on the same target — same paper, arXiv preprint vs journal version of the same paper, same-author republication / self-quote, mis-deduplicated id lineage, or wording / rounding artefact of one proposition? — If yes, `confirmed_equivalence`.
2. Does the visible condition difference between the two sides already have a known mediating explanation in the field? — If yes, `resolved_moderator`.
3. Can a checkable new question be articulated, naming a specific axis a future measurement or calculation would discriminate, with at least one applicable `hypothesized_cause` from §5? — If no, `non_generative`. If yes, **promote** and fill the structured fields per §4.

If the agent suspects a corpus-layer artefact (extraction error, unit-conversion mistake, sign-flip) but has no internal evidence, do **not** dismiss on that basis — flag the suspicion in `rationale` and let the pair pass through the gate normally.

For a cross-validation candidate:

1. Do the two sides collapse to the same underlying work (same paper, arXiv preprint vs journal version of the same paper, same-author republication, same-author self-quote of the same result, mis-deduplicated id lineage)? — If yes, `confirmed_equivalence`. Resolve this **before** the independence check, because "same work counted twice" is not the same situation as "two distinct works whose pathways happen to share construction".
2. Are the two pathways independent in any of the senses listed in §7 property 3? — If no, `trivially_dependent`.
3. Can `independence_basis` and `scientific_weight` both be articulated as non-empty sentences per §8? — If no, `non_generative`. If yes, **promote** and fill the structured fields per §9.

If the agent suspects a corpus-layer artefact but has no internal evidence, do **not** dismiss on that basis — flag the suspicion in `rationale` and let the pair pass through the gate normally.

## 13. Effect on the rendered graph

Only **promoted** pairs may appear in `evidence_graph.json.pair_relations[]` and be drawn as graph pair edges. Dismissed pairs are never rendered. This rule is part of the contract and is enforced by `run-folder-output-contract.md` §8.

## 14. Caps and ordering

The per-file top-10 cap defined in `$evidence-graph-synthesis` §2b applies to **promoted** pairs only. Dismissed pairs do not consume cap quota. Within `dismissed_pairs.json` itself there is no cap — the file is an audit of every candidate pair the agent considered and rejected, so all dismissals are recorded.
